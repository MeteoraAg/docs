Use CPI when your Solana program must call DLMM directly in Rust, for example to swap, add/remove liquidity, claim fees, or manage a DLMM position as part of a larger protocol instruction.

<Warning>
  The `lb_clmm` program source is not open sourced. Public CPI integrations should be built from the published IDL and validated against transactions generated by the official TypeScript SDK.
</Warning>

## Program ID

```rust
pub const DLMM_PROGRAM_ID: Pubkey =
    pubkey!("LBUZKhRxPF3XUpBCjp4YzTKgLccjZhTSDM9YuVaPwxo");
```

The public program ID is the same for mainnet and devnet.

## Recommended CPI Workflow

1. Build the equivalent transaction with the TypeScript SDK first.
2. Inspect the generated instruction account order and remaining accounts.
3. Generate or define your Rust CPI account structs from the published IDL.
4. Forward the same bin arrays, bitmap extension accounts, Token-2022 remaining accounts, and signer accounts through your program.
5. Keep slippage and output constraints in your own instruction arguments so users sign an explicit bound.
6. Test on devnet against real deployed DLMM pools before mainnet deployment.

## V2 Instructions

New integrations should use the v2 instruction family because it supports both SPL Token and Token-2022 accounts.

| Flow | Program instruction | SDK method to study first |
| --- | --- | --- |
| Exact-in swap | `swap2` | `pool.swap(...)` |
| Exact-out swap | `swap_exact_out2` | `pool.swapExactOut(...)` |
| Add liquidity by strategy | `add_liquidity_by_strategy2` | `initializePositionAndAddLiquidityByStrategy(...)`, `addLiquidityByStrategy(...)` |
| Add liquidity by weights | `add_liquidity_by_weight2` | `addLiquidityByWeight2(...)` |
| Remove liquidity | `remove_liquidity2`, `remove_liquidity_by_range2` | `removeLiquidity(...)` |
| Claim fees | `claim_fee2` | `claimSwapFee(...)`, `claimAllSwapFee(...)` |
| Claim rewards | `claim_reward2` | `claimLMReward(...)`, `claimAllLMRewards(...)` |
| Limit order | `place_limit_order`, `cancel_limit_order` | `placeLimitOrder(...)`, `cancelLimitOrder(...)` |
| Rebalance | `rebalance_liquidity` | `simulateRebalancePosition(...)`, `rebalancePosition(...)` |

## Remaining Accounts

Several v2 instructions take remaining accounts. The IDL exposes the instruction interface, while the SDK is the practical reference for how those accounts are assembled.

Typical remaining account groups include:

| Account group | Why it is needed |
| --- | --- |
| Transfer hook accounts | Required when token X, token Y, reward mints, or host fee transfers use Token-2022 transfer hooks. |
| Bin arrays | Required for every bin array touched by swaps, liquidity, claims, and limit orders. |
| Bitmap extension | Required when bin array indexes fall outside the default bitmap range. |
| Reward accounts | Required for reward claim and reward-aware liquidity updates. |

<Tip>
  The safest way to debug CPI account lists is to create the same operation with the SDK, log the instruction keys, and compare them with the accounts your program forwards to DLMM.
</Tip>

## Swap CPI Shape

This example is intentionally schematic. The exact generated type names depend on your IDL-to-Rust generation approach.

```rust
use anchor_lang::prelude::*;

#[derive(Accounts)]
pub struct SwapThroughDlmm<'info> {
    /// CHECK: DLMM program account.
    pub dlmm_program: AccountInfo<'info>,
    /// CHECK: Validated by the DLMM program.
    #[account(mut)]
    pub lb_pair: AccountInfo<'info>,
    /// CHECK: Optional bitmap extension, when required by the swap route.
    #[account(mut)]
    pub bin_array_bitmap_extension: AccountInfo<'info>,
    /// CHECK: DLMM reserve token account.
    #[account(mut)]
    pub reserve_x: AccountInfo<'info>,
    /// CHECK: DLMM reserve token account.
    #[account(mut)]
    pub reserve_y: AccountInfo<'info>,
    /// CHECK: User input token account.
    #[account(mut)]
    pub user_token_in: AccountInfo<'info>,
    /// CHECK: User output token account.
    #[account(mut)]
    pub user_token_out: AccountInfo<'info>,
    /// CHECK: Token X mint.
    pub token_x_mint: AccountInfo<'info>,
    /// CHECK: Token Y mint.
    pub token_y_mint: AccountInfo<'info>,
    /// CHECK: DLMM oracle account.
    #[account(mut)]
    pub oracle: AccountInfo<'info>,
    pub user: Signer<'info>,
    /// CHECK: Token program for token X.
    pub token_x_program: AccountInfo<'info>,
    /// CHECK: Token program for token Y.
    pub token_y_program: AccountInfo<'info>,
    /// CHECK: Memo program.
    pub memo_program: AccountInfo<'info>,
}
```

Your handler should construct the DLMM CPI call from the IDL-generated instruction interface, then call `.with_remaining_accounts(ctx.remaining_accounts.to_vec())` so bin arrays and Token-2022 accounts reach the DLMM program in the expected order.

## Off-Chain Reference Flow

Use the SDK to produce the account plan for the same swap:

```typescript
const swapYtoX = true;
const binArrays = await pool.getBinArrayForSwap(swapYtoX, 4);

const quote = pool.swapQuote(
  inAmount,
  swapYtoX,
  slippageBps,
  binArrays,
  false,
);

const tx = await pool.swap({
  inToken,
  outToken,
  inAmount: quote.consumedInAmount,
  minOutAmount: quote.minOutAmount,
  lbPair: pool.pubkey,
  user: user.publicKey,
  binArraysPubkey: quote.binArraysPubkey,
});

const DLMM_PROGRAM_ID = new PublicKey(
  "LBUZKhRxPF3XUpBCjp4YzTKgLccjZhTSDM9YuVaPwxo",
);

const dlmmIx = tx.instructions.find((ix) =>
  ix.programId.equals(DLMM_PROGRAM_ID),
);

console.log(dlmmIx?.keys.map((key) => ({
  pubkey: key.pubkey.toBase58(),
  isSigner: key.isSigner,
  isWritable: key.isWritable,
})));
```

Mirror this account order in your CPI instruction, including remaining accounts.

## Best Practices

| Check | Detail |
| --- | --- |
| Quote first | Use SDK quote methods before calling your program so users sign with explicit slippage and output constraints. |
| Pass current bin arrays | Swaps and liquidity updates fail if the required bin arrays are missing or out of order. |
| Handle Token-2022 | Include transfer hook accounts and use the correct token program for each mint. |
| Budget compute | DLMM instructions may need a high compute budget, especially bin array initialization, wide positions, and multi-bin liquidity. |
| Avoid stale state | Re-fetch pool state before quoting. `active_id`, fees, bin liquidity, and rewards can change between transactions. |
| Use the IDL contract | Do not rely on private implementation details that are not exposed through the public IDL or SDK behavior. |
