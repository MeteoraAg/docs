The DLMM on-chain program is `lb_clmm`. It stores liquidity in discrete price bins rather than a continuous curve. Each bin has its own token balances, liquidity supply, fee state, reward state, and optional limit order state.

## Core Model

| Concept | Program representation | Notes |
| --- | --- | --- |
| Pool | `LbPair` | Holds token mints, reserves, active bin, fee parameters, rewards, oracle, status, pool type, and Token-2022 flags. |
| Bin | `Bin` inside `BinArray` | A single price point. Each bin can contain token X, token Y, liquidity supply, fees, rewards, and limit order liquidity. |
| Bin array | `BinArray` | Fixed-size container for consecutive bins. Swaps and liquidity operations pass the bin arrays they may touch. |
| Bitmap | `LbPair.bin_array_bitmap` and `BinArrayBitmapExtension` | Tracks initialized bin arrays. The extension is used when bin array indexes overflow the default bitmap range. |
| Position | `PositionV2` plus optional dynamic data | Tracks an LP owner's bin range, liquidity shares, fee checkpoints, reward checkpoints, operator, lock release point, and fee owner. |
| Reward | `RewardInfo` and per-position `UserRewardInfo` | Up to two reward mints can be configured for liquidity mining pools. |
| Limit order | `LimitOrder` and dynamic `LimitOrderBinData` | Stores owner order amounts by bin, side, and age for pools that support limit orders. |
| Oracle | `Oracle` | Stores observations used for TWAP-like reads and dynamic fee/reference updates. |
| Token badge | `TokenBadge` | Admin-controlled allowlist for Token-2022 mints with sensitive extensions. |

## Price Bins

`active_id` is the current active bin. `bin_step` controls the price increment between adjacent bins.

```typescript
const price = dlmmPool.fromPricePerLamport(...)
const binId = dlmmPool.getBinIdFromPrice(price, true)
const activeBin = await dlmmPool.getActiveBin()
```

## Pool Types

| Pair type | Created by | Intended use |
| --- | --- | --- |
| `Permissionless` | `initialize_lb_pair` | Legacy permissionless SPL Token pools. |
| `PermissionlessV2` | `initialize_lb_pair2` | Current permissionless path with configurable fee presets, Token-2022 support, function type, and collect fee mode. |
| `Permission` | `initialize_permission_lb_pair` | Operator/admin-controlled launches with activation settings and lock behavior. |
| `CustomizablePermissionless` | `initialize_customizable_permissionless_lb_pair` / `initialize_customizable_permissionless_lb_pair2` | Creator-configured pools with optional activation point, creator on/off control, base fee, and Alpha Vault support. |

## Function Type

| Function type | Meaning |
| --- | --- |
| `LiquidityMining = 0` | Pool can initialize and fund reward emissions. |
| `LimitOrder = 1` | Pool is configured for limit order behavior. |


## Token-2022

`PermissionlessV2` and v2 liquidity/swap instructions support SPL Token and Token-2022 mints. The program stores token program flags for token X and token Y, and the SDK fetches transfer-hook extra account metas when creating a `DLMM` instance.

Permissionless Token-2022 support includes standard extensions such as transfer fees and metadata. Sensitive extensions, including active transfer hooks or permanent delegate behavior, may require an admin-created `TokenBadge`.

## Dynamic Positions

`PositionV2` has a fixed base layout and can append dynamic `PositionBinData` when a position grows beyond the default bin count. The program supports increasing or decreasing the position range without closing the account.

Some important limits are:

| Constant | Purpose |
| --- | --- |
| `MAX_BINS_PER_POSITION` / `POSITION_MAX_LENGTH` | Maximum supported position width. |
| `MAX_RESIZE_LENGTH` | Maximum resize step. |
| `MAX_BIN_LENGTH_ALLOWED_IN_ONE_TX` | SDK transaction chunking threshold for large liquidity operations. |

Use `quoteExtendPosition`, `increasePositionLength`, `decreasePositionLength`, `createExtendedEmptyPosition`, or rebalance helpers when managing wide positions.

## Compute And Account Loading

DLMM operations can touch many bin arrays. Swaps and liquidity operations must provide the bin arrays needed to cover the route or position range. `initialize_bin_array` initializes all bin prices and can require a high compute budgetso you can optionally add an explicit compute unit instruction when initializing bin arrays.
