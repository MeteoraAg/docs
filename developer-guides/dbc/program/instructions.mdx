The DBC program groups instructions by role. The TypeScript SDK exposes these through `client.partner`, `client.pool`, `client.creator`, `client.migration`, and `client.state`.

## Partner Instructions

| Instruction | SDK path | Purpose |
| --- | --- | --- |
| `create_config` | `client.partner.createConfig` | Create a partner config that controls pool fees, curve shape, migration settings, token behavior, and fee recipients. |
| `create_partner_metadata` | `client.partner.createPartnerMetadata` | Create metadata for a partner fee claimer. |
| `claim_trading_fee` | `client.partner.claimPartnerTradingFee` / `claimPartnerTradingFee2` | Claim partner trading fees from a virtual pool. |
| `claim_partner_pool_creation_fee` | `client.partner.claimPartnerPoolCreationFee` | Claim partner share of pool creation fees. |
| `partner_withdraw_surplus` | `client.partner.partnerWithdrawSurplus` | Withdraw partner quote surplus after migration when allowed. |

## Creator Instructions

## Swap Instructions

| Instruction | Modes | Notes |
| --- | --- | --- |
| `swap` | Exact in | Legacy/simple exact-in path. |
| `swap2` | Exact in, partial fill, exact out | Preferred for trading integrations because it emits richer swap data. |

`swap2` uses `SwapParameters2`:

| Field | Meaning |
| --- | --- |
| `amount_0` | Input amount for exact-in/partial-fill, or requested output amount depending on mode. |
| `amount_1` | Slippage bound: minimum output for exact-in/partial-fill, or maximum input for exact-out. |
| `swap_mode` | `0` exact in, `1` partial fill, `2` exact out. |

<Note>
  For rate limiter pools and first-swap-with-min-fee validation, the instruction sysvar account is required in remaining accounts. Use the SDK builders unless you are intentionally constructing raw instructions.
</Note>

## Recommended Transaction Construction

1. Use the TypeScript SDK transaction builders for public integrations.
2. Use `swapQuote` or `swapQuote2` before building swap transactions.
3. Keep slippage values explicit in user-signed transactions.
4. Simulate before sending, especially for migration and fee-claim flows.
5. If constructing raw instructions, compare account order against the IDL or SDK-generated instruction.
