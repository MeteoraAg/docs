Dynamic Fee Sharing is intentionally narrow: it is built to split fees safely between a small group of recipients. This page summarizes what is compatible with the product and when to use each path.

## Compatibility Overview

| Area | Supported |
| --- | --- |
| Recipient count | 2 to 5 recipients per fee vault |
| Share values | Non-zero positive shares for each recipient |
| Token standard | SPL Token and supported Token 2022 mints |
| Vault types | Non-PDA fee vault and PDA fee vault |
| Funding methods | Manual funding and supported claim-by-integration funding |
| Compatible products | Supported DAMM v2 and DBC fee-claim actions |

## Recipient Compatibility

A fee vault supports **2 to 5 recipients**.

Each recipient needs:

- A wallet address.
- A non-zero share value.
- A recipient index used when claiming.

Dynamic Fee Sharing is best for small, known recipient groups. It is not designed as a mass payout tool for hundreds or thousands of wallets.

<Warning>
  Recipient addresses and shares should be finalized before vault creation. The program stores the configured recipient list in the fee vault and does not expose a general edit-recipient flow.
</Warning>

## Token Compatibility

Dynamic Fee Sharing can distribute fees in one token mint per vault.

Supported token types include:

- Standard SPL Token mints.
- Token 2022 mints with supported extensions.

For Token 2022, the program supports mints that use these extensions:

- Transfer Fee Config
- Metadata Pointer
- Token Metadata

If a Token 2022 mint uses unsupported extensions, the vault rejects it.

<Info>
  If a project wants to split fees for multiple token mints, it should create one fee vault per token mint.
</Info>

## Vault Type Compatibility

Dynamic Fee Sharing supports two fee vault types.

| Vault Type | Compatible With | Best For |
| --- | --- | --- |
| Non-PDA fee vault | Manual fee funding | Standalone fee-sharing agreements. |
| PDA fee vault | Manual fee funding and supported claim-by-integration funding | Protocol-integrated fee sharing tied to a base account. |

### Non-PDA Fee Vault

A Non-PDA fee vault is flexible and standalone. It can receive manually funded fees and distribute them to recipients.

Use it when the source of fees is external to Dynamic Fee Sharing or when the team wants to manage funding manually.

### PDA Fee Vault

A PDA fee vault is deterministic. It is derived from a base account and token mint, which makes it useful for protocol integrations.

PDA fee vaults are required for funding by claiming supported fees because the vault needs to sign the compatible source-program claim action.

## Funding Compatibility

Dynamic Fee Sharing supports two funding paths.

### Manual Funding

Manual funding works for both Non-PDA and PDA fee vaults.

A funder transfers the fee token into the vault. The vault records the net amount and updates fee-per-share accounting.

Use manual funding when:

- Fees come from an external source.
- The team wants to batch payouts.
- The fee source is not on the supported claim-by-integration list.
- The vault should behave like a transparent distribution account.

### Funding by Claiming Fees

Funding by claiming fees works for PDA fee vaults and only for whitelisted source-program actions.

The vault calls a supported fee-claim action, receives the claimed fee tokens into its token vault, and then adds those claimed tokens to the fee-sharing accounting.

Use this when the fee vault is part of a supported Meteora integration and should collect claimable fees directly.

## Compatible Source Programs and Actions

Dynamic Fee Sharing currently whitelists supported actions from DAMM v2 and DBC.

### DAMM v2

Supported DAMM v2 claim actions include:

| Action | Product Meaning |
| --- | --- |
| Claim Position Fee | Claim trading fees from a DAMM v2 position into the fee vault. |
| Claim Reward | Claim DAMM v2 reward emissions into the fee vault. |

### Dynamic Bonding Curve (DBC)

Supported DBC claim and withdrawal actions include:

| Action | Product Meaning |
| --- | --- |
| Creator Withdraw Surplus | Route creator surplus from a completed DBC launch into the fee vault. |
| Partner Withdraw Surplus | Route partner surplus from a completed DBC launch into the fee vault. |
| Claim Creator Trading Fee | Route creator trading fees into the fee vault. |
| Claim Trading Fee | Route eligible DBC trading fees into the fee vault. |
| Withdraw Migration Fee | Route eligible migration fees into the fee vault. |

<Note>
  Dynamic Fee Sharing only supports claim-by-integration actions that are explicitly whitelisted by the program. Manual funding remains available for other fee sources as long as the token mint is supported.
</Note>

## Compatibility by Use Case

| Use Case | Recommended Setup |
| --- | --- |
| Standalone partner payout | Non-PDA vault with manual funding. |
| Treasury-funded recurring split | Non-PDA or PDA vault with manual funding. |
| DAMM v2 position fee sharing | PDA vault with supported DAMM v2 claim action. |
| DAMM v2 reward sharing | PDA vault with supported DAMM v2 reward claim action. |
| DBC creator or partner revenue split | PDA vault with supported DBC claim or withdrawal action. |
| Multi-token revenue sharing | Separate fee vault per token mint. |

## What Is Not a Fit

Dynamic Fee Sharing may not be the right product when:

- The recipient set changes frequently.
- There are more than 5 recipients.
- The token mint uses unsupported Token 2022 extensions.
- The fee source requires an unsupported cross-program claim action.
- The payout requires complex conditional logic instead of fixed shares.

For those cases, teams should either use manual operational flows or build a custom distribution layer around their requirements.
