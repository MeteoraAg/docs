Zap is Meteora's position action layer for turning multi-step AMM operations into cleaner user flows. It helps users move between tokens and liquidity positions without manually handling every swap, deposit, withdrawal, and account update themselves.

In plain English: Zap lets a user bring one asset, convert it into the right pool assets, and add liquidity. It also lets a user exit or harvest from a position, then immediately convert the output into the token they actually want.

<Info>
  Zap runs on the mainnet and devnet program `zapvX9M3uf5pvy4wRPAbQgdQsM1xmuiFnkfHKPvwMiz`.
</Info>

<Note>
  Zap is a helper product. It does not replace DAMM v2, DLMM, or Jupiter. It coordinates supported actions around those products so users can get a smoother position-management experience.
</Note>

## Why Zap Exists

Liquidity positions often require more than one transaction-level action.

A user may need to:

- Swap into the correct token mix before adding liquidity.
- Add liquidity to a DAMM v2 position.
- Create a DLMM position and deposit into a bin range.
- Rebalance an existing DLMM position around the current price.
- Remove liquidity, claim fees, or claim rewards, then convert the output token.
- Choose how much of a newly received token balance should be swapped.

Without Zap, frontends and users need to stitch those actions together manually. Zap gives Meteora products a shared helper layer that can track intermediate balances, apply supported swap routes, and complete the position action with fewer user-facing steps.

## What Zap Can Do

Zap supports two product directions: **Zap In** and **Zap Out**.

### Zap In

Zap In helps a user convert available tokens into the token mix needed for a position, then deposit those tokens into a supported Meteora AMM.

Supported product flows include:

- Zap into a DAMM v2 position.
- Zap into a new DLMM position.
- Zap into an existing DLMM position.
- Rebalance an existing DLMM position by depositing into a new bin range.

### Zap Out

Zap Out helps a user convert tokens received from another action, such as a removal, fee claim, or reward claim.

A typical Zap Out flow starts after a user's token account receives tokens. Zap calculates the balance increase, applies the chosen zap-out percentage, and swaps the selected amount through a supported route.

Supported swap paths include:

- DAMM v2 swap.
- DLMM swap.
- Jupiter v6 route.
- Jupiter v6 shared-account route.

## How Zap Fits Into Meteora

Zap sits beside Meteora's AMM products.

```text
User action
   |
   |-- Swap through DAMM v2, DLMM, or Jupiter
   |-- Track intermediate balances with a ledger
   |-- Add liquidity, rebalance, or convert outputs
   |
Supported Meteora position outcome
```

It is especially useful for frontends, integrators, and power users who want position actions to feel like a single guided flow instead of a manual sequence of swaps and deposits.

<AccordionGroup>
  <Accordion title="Liquidity providers">

  </Accordion>
  <Accordion title="Active position managers">
    Users managing DLMM ranges can rebalance around the active price without manually reconstructing every withdrawal, swap, and redeposit action.
  </Accordion>
</AccordionGroup>

## What Zap Is Best For

Zap is best for workflows where the user knows the desired outcome, but the path there requires multiple token and position operations.

Good examples:

- Add liquidity to SOL-USDC with only SOL.
- Add liquidity to a MET-USDC pool using SOL as the starting asset.
- Create a DLMM position and deposit into a chosen bin range.
- Rebalance an out-of-range DLMM position around the current active bin.
- Claim or withdraw from a position, then convert the output through DAMM v2, DLMM, or Jupiter.
- Exit a position into one preferred token instead of manually holding both sides.
